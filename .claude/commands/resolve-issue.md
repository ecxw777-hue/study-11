# GitHub 이슈 해결 커스텀 커맨드 (Solve GitHub Issue Custom Command)

## 사용 방법

GitHub 이슈 번호를 입력하면 이슈를 가져와 분석하고, 코드베이스 기반의 상세 해결 계획을 수립하여 사용자 검토를 받습니다.

## 프롬프트

다음 GitHub 이슈를 분석하고 해결 계획을 수립해주세요:

**이슈 번호:** #$ARGUMENT

아래 절차를 순서대로 수행하세요:

---

### 1단계: 이슈 가져오기

`gh issue view $ARGUMENT` 명령으로 이슈 정보를 가져오세요. 가져올 내용:

- 제목, 본문, 라벨, 담당자, 마일스톤
- 댓글이 있다면 함께 확인 (`gh issue view $ARGUMENT --comments`)

이슈를 가져올 수 없는 경우 사용자에게 알리고 중단하세요.

### 2단계: 이슈 이해

가져온 이슈를 요약하세요:

- **이슈 요약**: 무엇을 해야 하는지 1-2문장으로 정리
- **이슈 타입**: feature / bug / enhancement / refactor / docs / chore
- **수용 기준**: 이슈가 해결되었다고 볼 수 있는 조건들 (이슈 본문에서 추출하거나, 명시되지 않았다면 직접 정의)

### 3단계: 코드베이스 분석

이슈 해결에 필요한 코드베이스를 철저히 분석하세요:

- 이슈와 직접 관련된 파일들을 읽고 현재 구현 상태를 파악
- 수정이 필요한 파일과 그 이유를 식별
- 신규 생성이 필요한 파일이 있다면 명시
- 기존 패턴/컨벤션을 파악하여 일관성 유지 방안 확인
- 영향받는 다른 파일이나 기능이 있는지 확인 (사이드 이펙트 분석)

### 4단계: 해결 계획 수립

분석 결과를 바탕으로 아래 형식의 상세 계획을 작성하세요:

---

#### 이슈 요약

> 이슈 제목과 핵심 내용 요약

#### 현재 상태 분석

- 관련 코드의 현재 구현 상태
- 이슈가 발생한 원인 (버그인 경우) 또는 현재 누락된 부분 (기능인 경우)

#### 해결 전략

전체적인 접근 방식을 설명하세요:
- 왜 이 접근 방식을 선택했는지
- 고려했지만 선택하지 않은 대안이 있다면 그 이유

#### 구현 단계

각 단계를 순서대로 나열:

**단계 N: 작업 제목**
- 작업 내용: 구체적으로 무엇을 하는지
- 대상 파일: `path/to/file.tsx`
- 변경 사항: 추가/수정/삭제할 코드의 핵심 내용
- 주의 사항: 이 단계에서 조심해야 할 점

#### 영향 범위

- 수정 파일 목록 (경로 + 변경 유형)
- 사이드 이펙트 가능성
- 기존 기능에 대한 영향

#### 검증 방법

- 변경 후 확인해야 할 항목들
- 테스트 방법 (수동/자동)
- 엣지 케이스

---

### 5단계: 사용자 검토 요청

계획 작성이 완료되면 반드시 사용자에게 검토를 요청하세요:

- 계획 전체를 보여준 후 진행 여부를 확인
- 수정이 필요한 부분이 있는지 질문
- 사용자가 승인하면 구현을 시작하고, 수정 요청이 있으면 계획을 업데이트

---

**추가 지침**

- 이슈 본문이 모호한 경우, 가능한 해석을 나열하고 사용자에게 확인
- 코드베이스를 충분히 읽지 않은 상태에서 추측하지 않기
- 계획의 각 단계는 독립적으로 커밋 가능한 단위로 분리
- 사용자 승인 없이 절대 코드 수정을 시작하지 않기
